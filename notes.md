# Build Notes — Salla Merchant Operations Agent

---

## 1. Design Decisions

### Why a sequential pipeline instead of a single agent loop?

The task can be decomposed into four independent transformations with clear input/output contracts. A sequential pipeline is:

- **Easier to debug** — each stage writes its output to disk, so you can inspect `sentiment_report.json` before running the pricing stage
- **Resumable** — if Stage 2.3 fails, you don't re-run the LLM classifier
- **Testable** — each stage can be unit-tested in isolation with a fixed input file

A single agent loop with tool calls would add orchestration complexity without any benefit here since there are no conditional branches between stages (pricing always follows sentiment, report always follows pricing).

---

### Why Ollama / local LLM instead of OpenAI?

Three reasons:

1. **No credentials required** — the submission runs on any machine with Ollama installed without API keys or environment variables
2. **No latency cost at scale** — classifying 5,000 messages with a cloud API at ~300ms/call would take 25 minutes. With deduplication, 32 local calls take ~10 seconds
3. **Offline operation** — consistent with Salla's embedded/merchant-device context where internet may be unreliable

Trade-off: `llama3.2:3b` is less capable than GPT-4 on ambiguous messages. The rule-based fallback partially compensates, and the dedup strategy means even a mediocre model gives consistent results (the same template is only classified once).

---

### Why dedup before LLM calls?

The customer message dataset has a characteristic property: many customers send near-identical messages (copy-paste complaints, automated templates). In the sample data, 5,000 messages compressed to 32 unique templates.

Classifying all 5,000 would:
- Take ~25 minutes on local hardware
- Produce identical results for identical inputs
- Add no signal value

The dedup-then-broadcast pattern gives 99.4% fewer LLM calls with identical output quality. The only cost is the deduplication pass itself (O(n) hash set lookup).

---

### Why rule-based fallback instead of just failing?

A merchant running this at 6am to review overnight messages should not get a Python traceback because Ollama crashed. The rule-based classifier:

- Always produces a valid result in the correct schema
- Handles the most common patterns accurately (refund/return → Transactional Request, "stopped working" → Complaint)
- Logs every fallback event to `agent_trace.log` so the operator knows Ollama was unavailable

The fallback is not a second-class path — it's designed to be production-safe.

---

### Why HTML output for the report?

The deliverable spec says "easy to act on". Options considered:

| Format | Pro | Con |
|---|---|---|
| JSON | Machine-readable, easy to pipe | Requires a separate viewer; not scannable at a glance |
| Markdown | Readable in any editor | No colour, no visual hierarchy for urgency |
| HTML | Full colour, badges, responsive | Requires a browser |

HTML won because the report is merchant-facing, not developer-facing. A red pulsing alert banner communicates urgency better than `"severity": "high"` in JSON. The file is fully self-contained (inline CSS + JS, Google Fonts is the only external dependency and degrades gracefully offline).

## 2. Assumptions

### Catalog analysis

| Assumption | Why |
|---|---|
| Prices as English words ("ninety") are valid data, not errors | Manual data entry by non-technical merchants is common |
| `"unknown"` or `"??"` cost = intentionally missing | Merchants often don't track costs for all SKUs; flagging is safer than discarding |
| Duplicate = same normalised title + same price | Same name, different price = likely a different SKU or variant |
| Lowest product_id = canonical record | Earlier entries are more likely to be the original, reviewed listing |
| Abbreviations flagged but not auto-expanded | "blk" could mean "black" or "bulk" depending on context |

### Sentiment analysis

| Assumption | Why |
|---|---|
| Stats computed over unique message templates, not raw rows | 500 copies of the same message should count as one data point for sentiment analysis |
| Complaint rate displayed against raw total (5,000) | The merchant cares about inbox volume, not template count |
| Urgency fuzzy-deduped by fingerprint in the report | "where's my order??" and "where's my order????" are the same complaint |
| Summary prefix generated by rule ("Customer reports…") not LLM | LLM summaries of short messages were noisier and less consistent than a template |

### Pricing

| Assumption | Why |
|---|---|
| HC-1 margin floor = 15% | Standard retail minimum; configurable if Salla's merchants use different targets |
| Target margin = 35% | Reasonable e-commerce benchmark; the engine prices toward this, not above it |
| "Trending upward" approximated as today > 2× daily avg | True trend detection requires multi-day history; this is the best proxy with single-day data |
| Max single increase = 10% | Prevents the agent from making a large price change the merchant hasn't reviewed |

## 3. Bugs Found and Fixed During Development

This section documents the debugging process as evidence of production awareness.

### Bug 1: Duplicate urgency messages in the report
**Symptom:** The "High-Urgency Customer Messages" section showed the same message 10–15 times with minor variations ("??" vs "????", with/without "pls respond asap").

**Root cause:** The `build_report()` function collected all rows with `urgency=high` from the full 5,000-row classified list. Since 50 copies of "where's my order??" all get broadcast the same `urgency=high` classification, all 50 appeared in the list.

**Fix 1 (exact dedup):** Added a `seen_summaries` set to skip exact duplicates.

**Fix 2 (fuzzy dedup):** Exact dedup still missed near-duplicates. Added a fingerprint function that strips punctuation, collapses whitespace, removes trailing filler phrases, and truncates to 60 chars. "where's my order???? pls respond asap" and "where's my order?? it said delivery 2 days ago" now share the same fingerprint and collapse to one entry.

---

### Bug 2: Inflated complaint rate (50% instead of ~33%)
**Symptom:** After the first dedup fix, complaint rate jumped from 33% to 50% and complaint count showed 2,499.

**Root cause:** `build_category_stats()`, `build_product_stats()`, and `detect_anomalies()` all iterated the full 5,000-row classified list. Since rows were classified by unique template, 500 copies of "blender stopped working" (category=Complaint) added 500 to the complaint count. The actual number of distinct complaint *types* was 14.

**Fix:** All aggregation functions now deduplicate by message text before counting. Each unique template contributes exactly once to category counts, product stats, and the complaint rate.

---

### Bug 3: 0% negative rate on products with hundreds of complaints
**Symptom:** Blender showed "502 complaints (0% negative)" and Cook Set showed "497 complaints (0% negative)".

**Root cause:** The LLM (or rule-based fallback) classified "the blender stopped working after 3 uses" as `category=Complaint, sentiment=neutral` — the complaint signals fired but the negative sentiment keywords were absent. With 500 duplicate rows all carrying `sentiment=neutral`, `negative=0` despite `complaints=500`.

**Fix:** `build_product_stats()` now deduplicates by message text before computing neg_rate. The negative rate reflects the fraction of *distinct complaint types* with negative sentiment — an accurate measure of whether customers are expressing frustration vs. just reporting a factual problem.

---

### Bug 4: "across 32 messages" in the complaint spike alert
**Symptom:** Alert read "Complaint rate 43.8% across 32 messages" — confusing because the merchant's inbox actually has 5,000 messages.

**Root cause:** After the aggregation dedup fix, `total = len(unique_classified) = 32`. The alert message used this deduplicated count as the denominator string.

**Fix:** Separated two concerns: (a) rate is computed over unique templates for accuracy, (b) the displayed denominator is the raw inbox total (5,000) and the complaint count is scaled back proportionally so the alert reads "43.8% across 5,000 messages".

## 4. Trade-offs and Known Limitations

### Sentiment accuracy at 3B parameters
`llama3.2:3b` is fast and local but makes classification errors on ambiguous messages. Example: "the cook set is good but the lids don't fit perfectly" — this is a complaint but the word "good" may push the model toward neutral sentiment. The rule-based fallback has the same limitation.

**Production fix:** Use a fine-tuned classification model on e-commerce data, or a larger model (7B+). Alternatively, add a human review queue for low-confidence classifications.

### Single-day snapshot
The pipeline processes one day's data at a time. HC-2's "trending upward" block uses a ratio proxy rather than real trend data.

**Production fix:** Persist `sentiment_report.json` across runs with a date key. The pricing engine reads the last 7 days and computes a proper trend slope.

### No real-time processing
The pipeline is batch-mode: run once, get a report. High-urgency messages from 3am won't be flagged until the next morning run.

**Production fix:** Run the sentiment pipeline on a 1-hour cron. Only re-classify new messages (delta since last run) to keep LLM costs low.

### Product name matching is manual
`PRODUCT_ALIASES` in `pricing_recommendations.py` maps short names ("Blender") to catalog titles ("Portable Blender"). Adding a new product requires updating this dict.

**Production fix:** Use fuzzy string matching (e.g. `rapidfuzz`) or let the LLM extract the full product name from the catalog rather than a short form.

## 5. Edge Cases (Bonus)

### What happens if the CSV has no cost column?
`clean_cost()` treats missing or malformed cost values as `None` with a `MISSING_COST` flag. The product is kept in the catalog. In the pricing engine, missing cost → `HOLD` with explanation "HC-1 cannot be verified — enter the cost so margin analysis can proceed." No crash, no silent wrong recommendation.

### What if all messages are duplicates of one template?
The dedup pipeline handles this correctly: 1 unique text → 1 LLM call → broadcast to all rows. Stats will show 1 unique template, 5,000 raw messages. The `REPEATED_MESSAGES` anomaly alert will fire with the full count. The complaint rate is computed as either 0% or 100% (one template, one category).

### What if Ollama returns valid JSON but with wrong field values?
`classify_message()` validates every field against `VALID_CATEGORIES`, `VALID_SENTIMENTS`, and `VALID_URGENCIES` after receiving the Ollama response. Any out-of-spec value is replaced with the safe default (`"Inquiry"`, `"neutral"`, `"low"`). The bad value is not silently passed through.

### What if pricing_context.csv is missing?
`load_pricing_context()` checks for file existence before opening. If absent, it returns `{}` and logs a note. The pricing engine continues with catalog cost/price only — `market_trend` defaults to `"unknown"`, competitor fields to `None`. Recommendations are still generated; they just lack market-positioning context.

### What if a product has no complaints but a rising complaint count trend?
The `COMPLAINT_TREND_RATIO` check only fires if `avg_daily_complaints > 0`. A product with zero historical complaints and zero today produces `0 > 0 * 2.0 = False` — no false-positive block. A product with 1 avg daily complaint and 3 today produces `3 > 1 * 2.0 = True` — correctly blocked.

### What if the same product appears under multiple name variants in the sentiment report?
`resolve_sentiment_signals()` iterates all keys in the sentiment report's `top_products_by_complaints` dict and checks each against `PRODUCT_ALIASES`. Multiple matches (e.g. both "Blender" and "Portable Blender" present) are summed together into a single signal object for the catalog product. No double-counting occurs because aliases always map to the same canonical title.
